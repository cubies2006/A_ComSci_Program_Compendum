At the core of mobile game usability: the pause menu

Introduction

This article discusses issues related to the pause feature in mobile games and provides some guidelines for improving the overall usability and game experience of mobile games. The examples used in this article are based on the S60 and Series 40 platforms.

In a mobile context, users face interruptions and distractions such as an incoming call from home, a new message from a friend, or getting off at the bus stop. Thus, providing an easy and fast way to pause the game and being able to continue it later is an essential feature of a mobile game. The pause menu is used when the player switches context, for example when adjusting sound levels or pausing the game for a few seconds. The game developer’s understanding of the mobile context is often visible from the pause menu implementation; it is the essence of mobile game usability.

Tip: To quickly evaluate a mobile game’s quality, take a look at its pause menu.

The following guidelines will help developers in designing usable pause features.

To pause or not to pause?

   

Some games might not seem to need a pause menu at all. There does not seem to be any point for a pause menu in strategy games that do not contain real-time action. However, users still need a pause feature — the ability to interrupt the game quickly and continue it later.

In Series 40 devices, the quickest way to quit a game is by pressing the End key. This should save the current situation and quit the game. When the player later opens the game again, the first option should be “continue game.” So, even strategy games need a pause feature, but it can be implemented as automatic saving and loading.

Automatic saving and loading can be used as a Pause feature in Series 40 games.

S60 games are similar but a bit more complex because of the platform’s multitasking features. The user can press the Application key (or also the End key prior to S60 2nd Edition, Feature Pack 3) to quickly exit the game. The game should then be paused, and also saved. The implementation of multitasking in S60 works well with interruptions — even so that users often do not even notice it. Unfortunately, it cannot be relied on as an “automatic save game feature”; users could power off the device, and thus lose the game. The same goes for other multitasking platforms, such as Series 80.

Tip: Advanced S60 users may know that a game can be paused by opening the Power key menu and selecting keypad lock. If this feature could be useful for pausing your game, it should be mentioned in the game instructions. This feature could be useful for always-on games, which still need to be physically paused, even if the game is not paused.

The context change in a S60 game needs to be thought out too. Users may have had sounds on when they switched the game to the background, but when they continue the game, they might be in a situation where they prefer the silent mode. This is one reason more for providing sound settings in Pause menus. This also means that games should check the current profile when continuing the game from a paused state.

Pause menu or main menu?

  

There are two recommended ways for implementing the pause feature. The first is having a separate pause menu. This is recommended for more complex games. The other is combining the game main menu and pause menu to a single menu. This is recommended for very simple games.

Disable sounds and music

A game is often paused when something happens in the player’s surroundings. Pressing the pause button should automatically stop all the sounds, music, vibrations, and other animations. On platforms that support multitasking, it is essential that the game does not consume resources while it is open in the background. Some bad examples include games that consume memory or processing power while in the background. Some games have even played sounds when in the background, and novice users might not know how to close the application, except by powering off the whole device. Minimize the use of system resources when paused.

Actions in the pause menu

The default action in a pause menu is, of course, resuming the game.

The player should be able to resume the game with the same key with which the game was paused. Using different keys for pause and resume is not recommended and not very intuitive for the player. This is usually conveniently implemented by having “Resume game” as the first option in the pause menu.

In addition to just pausing and resuming, provide options that might be useful for the player. In some games, it might be a good idea to add options for other frequently needed actions, such as quickly restarting the level or exiting the game. Players should also be able to read Help topics without leaving the game. It is a good idea to add Help topics to the pause menu. This way, players don’t need to memorize all the instructions before trying the game. This is an especially useful feature in complex games.

Resuming the game

In action games, using the pause menu might get a player’s character killed; typically, when the user continues the action, the user might get surprised, or might not yet be prepared for the action. Some common solutions for this include displaying the game situation in the background of the pause menu, or not having the game action start instantly after the “resume” key press.

To sum up:

  • A mobile game needs to be paused frequently and quickly.

  • Allow players to continue the game later, without getting the player’s character killed.

  • Provide Help; it might be needed during the game.

  • Provide sound and vibration settings; they are needed when the user changes context.

  • Minimize the use of system resources when paused.
